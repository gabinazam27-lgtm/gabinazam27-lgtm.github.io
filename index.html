ARBRES I. Arbres binaires

1.1 Definitions

Definition

Un arbre binaire est une structure ou chaque noeud a au plus deux fils : sous-arbre gauche (sag) et sous-arbre droit (sad). Le noeud sans pere est la racine. Un noeud sans fils est une feuille.

Definition recursive

Un arbre binaire est soit :

  • Vide (note None en Python) -- c'est le cas de base
  • Un noeud contenant une valeur, un sous-arbre gauche (lui-meme un arbre binaire) et un sous-arbre droit (lui-meme un arbre binaire)

Cette definition recursive est fondamentale : elle justifie que tous les algorithmes sur les arbres soient naturellement recursifs. Si l'arbre est vide, on retourne un cas de base. Sinon, on traite la racine et on appelle recursivement sur sag et sad.

Vocabulaire
  • Racine -- noeud au sommet (pas de pere)
  • Feuille -- noeud sans fils (sag = sad = None)
  • Hauteur -- nb de niveaux - 1
  • Profondeur -- distance d'un noeud a la racine
  • Taille -- nombre total de noeuds
  • Noeud interne -- noeud qui a au moins un fils

Arbre binaire : racine = 5, feuilles = 15, 25, 30

Types d'arbres binaires

Arbre complet (parfait en anglais)

Tous les niveaux sont completement remplis. Un arbre complet de hauteur h a exactement 2h+1 - 1 noeuds. C'est la forme la plus compacte possible.

Arbre parfait (complet en anglais)

Tous les niveaux sont remplis sauf eventuellement le dernier, qui est rempli de gauche a droite. C'est la forme utilisee dans les tas.

Arbre degenere

Chaque noeud a au plus un fils. L'arbre ressemble a une liste chainee. Hauteur = n - 1 (pire cas). Toutes les operations deviennent O(n).

Arbre equilibre

La hauteur des sous-arbres gauche et droit de chaque noeud differe d'au plus 1. Hauteur = O(log n). Les AVL sont des arbres equilibres.

Relations hauteur / taille

Pour un arbre binaire a n noeuds et de hauteur h :

  • Hauteur minimale : h = floor(log2(n)) -- arbre complet/parfait, le plus compact
  • Hauteur maximale : h = n - 1 -- arbre degenere (liste chainee)
  • Donc : log2(n) ≤ h ≤ n - 1
  • Nombre de noeuds : h + 1 ≤ n ≤ 2h+1 - 1

Exemple : avec 7 noeuds, la hauteur va de 2 (arbre complet) a 6 (degenere).

def largeur(a):

    file = [a]

    if a is None: return -1

    return nb_feuilles(a['sag']) + nb_feuilles(a['sad'])

1.4 ABR (Arbre Binaire de Recherche)

Propriete

Pour chaque noeud : toutes les valeurs du sous-arbre gauche < racine < toutes les valeurs du sous-arbre droit. Recherche en O(log n) si equilibre, O(n) si degenere.

Exemple d'ABR avec recherche de la valeur 13

ABR : recherche de 13. Chemin en rouge : 10 → 15 → 13 (3 comparaisons)

Infixe d'un ABR = valeurs triees

L'infixe (GRD) de cet ABR donne : 1, 3, 5, 6, 7, 10, 13, 15, 20 -- ordre croissant garanti. C'est une propriete fondamentale de l'ABR : le parcours infixe visite d'abord tout le sous-arbre gauche (valeurs plus petites), puis la racine, puis le sous-arbre droit (valeurs plus grandes). Cela produit toujours un ordre trie croissant.

def recherche_abr(a, val):
    if a is None: return False
    if val == a['valeur']: return True
    if val < a['valeur']: return recherche_abr(a['sag'], val)
    return recherche_abr(a['sad'], val)

def insertion_abr(a, val):
    if a is None: return {'valeur': val, 'sag': None, 'sad': None}
    if val < a['valeur']: a['sag'] = insertion_abr(a['sag'], val)
    elif val > a['valeur']: a['sad'] = insertion_abr(a['sad'], val)
    return a

Minimum et Maximum dans un ABR

Principe

Le minimum d'un ABR se trouve en allant toujours a gauche jusqu'a la feuille la plus a gauche. Le maximum se trouve en allant toujours a droite. Complexite : O(h) ou h est la hauteur de l'arbre.

def minimum_abr(a):
    # Le minimum est le noeud le plus a gauche
    if a is None: return None
    if a['sag'] is None: return a['valeur']
    return minimum_abr(a['sag'])

def maximum_abr(a):
    # Le maximum est le noeud le plus a droite
    if a is None: return None
    if a['sad'] is None: return a['valeur']
    return maximum_abr(a['sad'])

Attention

Memes valeurs {1,2,3,4,5} mais l'ordre d'insertion change tout. A gauche : insertion 3,1,5,2,4 donne un arbre equilibre, recherche O(log n). A droite : insertion 1,2,3,4,5 (ordre croissant) donne un arbre degenere en liste chainee, recherche O(n). C'est pourquoi en pratique on utilise des ABR equilibres (AVL, arbres rouge-noir) pour garantir h = O(log n).

1.5 Tas (Heap)

Definition

Tas max : arbre binaire complet ou chaque pere est superieur ou egal a ses fils. Le maximum est toujours a la racine. Un tas min est l'inverse (pere ≤ fils, minimum a la racine).

Correspondance arbre / tableau

Un tas se stocke dans un tableau (sans pointeurs). C'est possible car l'arbre est complet : il n'y a pas de "trous". Pour un noeud a l'indice i (en commencant a 0) :

Formules d'indexation
  • Racine = indice 0
  • Fils gauche de i = 2*i + 1
  • Fils droit de i = 2*i + 2
  • Pere de i = (i - 1) // 2
  • Dernier noeud interne = n//2 - 1
Exemple concret

Tableau : [50, 30, 40, 10, 20, 35, 5, 8]

  • Racine : T[0] = 50
  • Fils gauche de T[0] : T[1] = 30
  • Fils droit de T[0] : T[2] = 40
  • Pere de T[5] : T[(5-1)//2] = T[2] = 40
  • Fils de T[3]=10 : T[7]=8, T[8] n'existe pas

Tas max : vue arbre (haut) et vue tableau (bas). Les lignes pointillees montrent la correspondance.

Descendre (Sift-Down / Heapify)

Pour maintenir la propriete du tas apres modification, on tamise l'element vers le bas en l'echangeant avec son plus grand fils tant qu'il est plus petit. C'est l'operation fondamentale du tas.

Principe de descendre

On compare le noeud a l'indice i avec ses deux fils. Si l'un des fils est plus grand, on echange le noeud avec le plus grand des deux fils, puis on continue recursivement a la nouvelle position. On s'arrete quand le noeud est plus grand que ses deux fils, ou quand on atteint une feuille.

def descendre(T, n, i):
    # Tamise T[i] vers le bas dans un tas de taille n
    plus_grand = i
    g = 2 * i + 1    # fils gauche
    d = 2 * i + 2    # fils droit
    if g < n and T[g] > T[plus_grand]:
        plus_grand = g
    if d < n and T[d] > T[plus_grand]:
        plus_grand = d
    if plus_grand != i:
        T[i], T[plus_grand] = T[plus_grand], T[i]
        descendre(T, n, plus_grand)  # continuer a descendre

Construire un tas

Pourquoi partir de n//2 - 1 ?

Les elements d'indice n//2 a n-1 sont des feuilles : ils n'ont pas de fils, donc ils satisfont deja la propriete du tas (trivialement). On commence donc par le dernier noeud interne (n//2 - 1) et on remonte vers la racine, en appliquant descendre a chaque noeud. Complexite : O(n) (et non O(n log n) car les noeuds proches des feuilles descendent peu).

def construire_tas(T):
    n = len(T)
    # Partir du dernier noeud interne, remonter vers la racine
    for i in range(n // 2 - 1, -1, -1):
        descendre(T, n, i)

Tri par tas (Heapsort)

Principe du tri par tas

Le tri par tas fonctionne en 2 phases :

  1. Construire le tas max a partir du tableau (O(n))
  2. Extraire le maximum repetitivement : echanger T[0] (le max) avec le dernier element, reduire la taille du tas de 1, puis descendre(T, nouvelle_taille, 0) pour restaurer la propriete. Repeter n-1 fois.

A chaque etape, le maximum est place a sa position finale (fin du tableau). Complexite totale : O(n log n), en place, non stable.

def tri_par_tas(T):
    n = len(T)
    construire_tas(T)            # Phase 1 : construire le tas max
    for i in range(n - 1, 0, -1):
        T[0], T[i] = T[i], T[0]  # Phase 2 : echanger max avec dernier
        descendre(T, i, 0)        # restaurer le tas (taille reduite)
Tas != ABR

Ne pas confondre : un tas garantit seulement pere ≥ fils (pas de relation gauche/droite). Un ABR garantit gauche < racine < droite. Un tas ne donne pas un parcours trie, un ABR si (infixe).

1.6 Arbre d'expression arithmetique

Une expression comme (3 - (5+4)*(5-1)) se represente en arbre : les operateurs sont les noeuds internes, les nombres sont les feuilles.

Arbre d'expression pour (3 - (5+4)*(5-1)) = 3 - 36 = -33

Parcours de l'arbre d'expression

Prefixe : -, 3, *, +, 5, 4, -, 5, 1  (notation polonaise)

Infixe : 3, -, 5, +, 4, *, 5, -, 1  (lecture naturelle, sans parentheses)

Postfixe : 3, 5, 4, +, 5, 1, -, *, -  (notation polonaise inverse -- calculatrices HP)

1.7 Trie (arbre de prefixes)

Definition

Un trie est un arbre n-aire ou chaque arete represente un caractere. On l'utilise pour stocker un dictionnaire de mots. Recherche en O(longueur du mot), independant du nombre de mots stockes.

Pourquoi O(longueur du mot) ?

Pour chercher un mot de longueur m, on descend dans le trie en suivant une branche par caractere. A chaque niveau, on fait une seule comparaison (ou un acces dictionnaire en O(1)). On descend exactement m niveaux. Donc la complexite est O(m), totalement independante du nombre de mots dans le trie ! C'est bien meilleur qu'une recherche dans une liste triee (O(m log n)) ou qu'un parcours lineaire (O(n * m)).

Trie contenant {a, an, anti, art, va}. Les noeuds verts marquent une fin de mot.

Algorithme de recherche dans un trie

Pour chercher un mot, on descend dans le trie caractere par caractere. A chaque niveau, on cherche le fils correspondant au caractere courant. Si on ne le trouve pas, le mot n'est pas dans le trie. Si on arrive au bout du mot, on verifie que le noeud courant est marque comme fin de mot.

# Representation : chaque noeud = {"fin": bool, "fils": {lettre: noeud}}

def recherche_trie(racine, mot):
    """Cherche si mot est dans le trie. O(len(mot))"""
    noeud = racine
    for c in mot:
        if c not in noeud["fils"]:
            return False   # caractere non trouve, mot absent
        noeud = noeud["fils"][c]
    return noeud["fin"]       # True seulement si c'est un mot complet

def inserer_trie(racine, mot):
    """Insere un mot dans le trie. O(len(mot))"""
    noeud = racine
    for c in mot:
        if c not in noeud["fils"]:
            noeud["fils"][c] = {"fin": False, "fils": {}}
        noeud = noeud["fils"][c]
    noeud["fin"] = True
Attention a la difference prefixe / mot complet

Chercher "an" dans le trie {a, an, anti, art, va} retourne True car "an" est marque comme fin de mot. Mais chercher "ant" retourne False : le noeud 't' apres "an" existe (pour "anti") mais n'est pas marque comme fin de mot.

GRAPHES II. Graphes

2.1 Vocabulaire

Analogies concretes

Graphe oriente = reseau de rues a sens unique. Chaque arc a une direction : on peut aller de A vers B mais pas forcement de B vers A. Exemple : Twitter (suivre quelqu'un est unilateral).

Graphe non oriente = reseau de routes bidirectionnelles. Chaque arete relie les deux sommets dans les deux sens. Exemple : Facebook (l'amitie est bilaterale).

  • Sommet -- element du graphe
  • Arc -- lien oriente / Arete -- lien non oriente
  • Voisin -- sommet directement relie
  • Boucle -- arc vers soi-meme
  • Degre -- nombre d'aretes incidentes a un sommet
  • Connexe -- en un seul bloc
  • Complet -- tout relie a tout
  • Pondere -- poids sur les aretes
  • Non oriente → matrice symetrique
  • Chemin -- suite de sommets relies par des aretes
  • Cycle -- chemin qui revient au sommet de depart
Nombre maximal d'aretes

Graphe non oriente (sans boucle) : chaque paire de sommets peut avoir au plus 1 arete. Il y a C(n,2) = n(n-1)/2 paires. Exemple : 4 sommets → max 6 aretes.

Graphe oriente (sans boucle) : chaque paire ordonnee peut avoir 1 arc. Il y a n(n-1) paires ordonnees = n(n-1) arcs max. Exemple : 4 sommets → max 12 arcs.

Graphe oriente (arcs avec fleches)

Graphe oriente a 4 sommets : les fleches indiquent le sens des arcs

Representations du graphe oriente

Matrice d'adjacence (oriente)

M[i][j] = 1 si arc de i vers j. Non symetrique en general.

#     0  1  2  3
M = [[0,1,1,0],  # 0 → 1,2

     [0,0,0,1],  # 2 → 3

Liste d'adjacence (oriente)

L[i] = sommets atteignables depuis i (successeurs).

L = [[1,2],  # 0 → 1,2
     [3],     # 1 → 3
     [3],     # 2 → 3
     [0]]     # 3 → 0

Graphe non oriente (aretes sans fleches)

Graphe non oriente a 4 sommets : pas de sens sur les aretes

Representations du graphe non oriente

Matrice d'adjacence (non oriente)

M[i][j] = M[j][i] = 1 si arete {i,j}. Toujours symetrique.

#     0  1  2  3
M = [[0,1,1,1],  # 0 -- 1,2,3
     [1,0,0,1],  # 1 -- 0,3
     [1,0,0,1],  # 2 -- 0,3
     [1,1,1,0]]  # 3 -- 0,1,2
Liste d'adjacence (non oriente)

L[i] = tous les voisins de i. Chaque arete apparait 2 fois.

L = [[1,2,3],  # 0 -- 1,2,3
     [0,3],     # 1 -- 0,3
     [0,3],     # 2 -- 0,3
     [0,1,2]]   # 3 -- 0,1,2

Graphe pondere

Representation des poids

Matrice ponderee : M[i][j] = poids de l'arete entre i et j. On utilise 0 (ou inf) pour indiquer l'absence d'arete. Attention : si 0 est un poids valide, on utilise float('inf') a la place.

Liste d'adjacence ponderee : L[i] = liste de tuples (voisin, poids). Exemple : L[0] = [(1, 6), (2, 9)] signifie arete 0-1 de poids 6, arete 0-2 de poids 9.

Graphe pondere a 7 sommets. Poids en rouge sur chaque arete.

Quand utiliser matrice vs liste ?

Matrice d'adjacence : ideale pour les graphes denses (beaucoup d'aretes, proche du complet). Acces O(1) pour tester si une arete existe. Memoire O(n^2).

Liste d'adjacence : ideale pour les graphes creux (peu d'aretes). Parcours des voisins rapide. Memoire O(n + m) ou m = nombre d'aretes.

En pratique, la plupart des graphes reels (web, reseaux sociaux, routes) sont creux, donc on prefere la liste d'adjacence.

2.2 Conversions et representations Python

Conversions

Pleine → creuse : pour chaque ligne i, on garde les indices j ou M[i][j] != 0. Creuse → pleine : on cree une matrice de zeros et on met 1 aux positions indiquees.

def pleine_vers_creuse(M):
    return [[j for j in range(len(M)) if M[i][j] != 0] for i in range(len(M))]

def creuse_vers_pleine(L):
    n = len(L)
    M = [[0]*n for _ in range(n)]
    for i in range(n):
        for j in L[i]: M[i][j] = 1
    return M

2.3 DFS vs BFS -- Parcours en profondeur et en largeur

DFS (Depth-First Search)
  • Utilise une pile (LIFO)
  • .pop() -- dernier entre, premier sorti
  • Explore le plus profond d'abord
  • Detection de cycles, composantes connexes
BFS (Breadth-First Search)
  • Utilise une file (FIFO)
  • .pop(0) -- premier entre, premier sorti
  • Explore niveau par niveau
  • Plus court chemin (non pondere)

Code simple (sans chemin)

def dfs(graphe, depart):
    pile = [depart]
    visite = []
    while pile:
        s = pile.pop()           # PILE -- dernier entre, premier sorti
        if s not in visite:
            visite.append(s)
            for v in graphe[s]:
                if v not in visite:
                    pile.append(v)
    return visite

def bfs(graphe, depart):
    file = [depart]
    visite = []
    while file:
        s = file.pop(0)          # FILE -- premier entre, premier sorti
        if s not in visite:
            visite.append(s)
            for v in graphe[s]:
                if v not in visite:
                    file.append(v)
    return visite
A retenir

Seule difference DFS/BFS : .pop() (pile) vs .pop(0) (file). Le reste est identique.

Code complet avec suivi de chemin (du TP)

Version avec chemin

La pile/file stocke des tuples (sommet, chemin) ou chemin est la liste des sommets parcourus pour arriver a ce sommet. Cela permet de retrouver le chemin depuis le depart.

def dfs_chemin(graphe, depart, arrivee):
    """DFS avec suivi de chemin. Retourne le chemin depart -> arrivee."""
    pile = [(depart, [depart])]     # (sommet, chemin pour y arriver)
    visite = []
    while pile:
        s, chemin = pile.pop()    # PILE : dernier entre, premier sorti
        if s == arrivee:
            return chemin           # Trouve !
        if s not in visite:
            visite.append(s)
            for v in graphe[s]:
                if v not in visite:
                    pile.append((v, chemin + [v]))
    return None                   # Pas de chemin

def bfs_chemin(graphe, depart, arrivee):
    """BFS avec suivi de chemin. Retourne le PLUS COURT chemin."""
    file = [(depart, [depart])]     # (sommet, chemin pour y arriver)
    visite = []
    while file:
        s, chemin = file.pop(0)   # FILE : premier entre, premier sorti
        if s == arrivee:
            return chemin           # Trouve (et c'est le plus court !)
        if s not in visite:
            visite.append(s)
            for v in graphe[s]:
                if v not in visite:
                    file.append((v, chemin + [v]))
    return None                   # Pas de chemin

2.4 Kruskal -- Arbre Couvrant Minimum (ACM)

Objectif

Trouver le sous-ensemble d'aretes qui connecte tous les sommets avec le poids total minimum, sans creer de cycle. Le resultat est un arbre (n-1 aretes pour n sommets).

Algorithme en 4 etapes

  1. Trier toutes les aretes par poids croissant
  2. Attribuer un numero de composante a chaque sommet (au depart, chaque sommet a son propre numero : comp[0]=0, comp[1]=1, comp[2]=2, ...)
  3. Pour chaque arete (u, v) dans l'ordre croissant :
    • Si comp[u] == comp[v] : rejeter (creerait un cycle -- meme composante)
    • Si comp[u] != comp[v] : retenir l'arete, puis unifier les numeros (tous les sommets qui avaient comp[v] prennent comp[u])
  4. S'arreter quand on a retenu n-1 aretes (ou quand toutes sont traitees)
Union-Find par numeros de composante

L'idee est simple : chaque sommet porte un numero de composante. Au depart, chaque sommet est dans sa propre composante (comp[i] = i). Quand on retient une arete (u, v), on fusionne les composantes : tous les sommets qui avaient le meme numero que v recoivent maintenant le numero de u.

Find = simplement lire comp[i] (O(1)).

Union = parcourir tout le tableau et remplacer l'ancien numero par le nouveau (O(n) dans le pire cas).

Cette technique est celle des slides du cours. Il existe des versions plus efficaces (Union-Find avec rang + compression de chemin en O(alpha(n))) mais elles ne sont pas au programme.

2.5 Dijkstra -- Plus courts chemins

Hors programme CC

Dijkstra est au programme du partiel final, pas du controle continu. Le comprendre donne un avantage.

Principe

Trouver le plus court chemin depuis un sommet source vers tous les autres, dans un graphe pondere a poids positifs uniquement. On maintient deux tableaux : Dist[i] (distance minimale connue depuis la source) et Traj[i] (predecesseur sur le chemin optimal).

Poids positifs uniquement !

Dijkstra ne fonctionne pas avec des poids negatifs. Si un poids est negatif, on pourrait trouver un raccourci en passant par une arete negative apres avoir deja traite un sommet, ce qui fausserait les distances. Pour les poids negatifs, il faut utiliser l'algorithme de Bellman-Ford.

Algorithme detaille

  1. Initialiser Dist[source] = 0, tous les autres a l'infini. Traj[i] = None pour tous.
  2. Tous les sommets sont "non traites" (dans un ensemble restants).
  3. Repeter tant qu'il reste des sommets non traites :
    • Choisir le sommet non traite u avec le plus petit Dist[u]
    • Marquer u comme traite (le retirer de restants)
    • Pour chaque voisin v non traite de u : si Dist[u] + poids(u,v) < Dist[v], alors mettre a jour Dist[v] = Dist[u] + poids(u,v) et Traj[v] = u (relaxation)
  4. S'arreter quand tous les sommets sont traites

Lecture du chemin (remonter Traj)

Pour trouver le chemin vers E : Traj[E]=D, Traj[D]=B, Traj[B]=A. On remonte et on inverse : A → B → D → E (cout = 4).

Pour trouver le chemin vers C : Traj[C]=A. Donc A → C (cout = 4).

Pour trouver le chemin vers D : Traj[D]=B, Traj[B]=A. Donc A → B → D (cout = 3).

COMPLEXITE III. Complexite algorithmique

3.1 Notation de Landau (Grand O)

T(n) = O(f(n)) ⇔ ∃ c, n₀ : ∀n > n₀, T(n) ≤ c x f(n)
Explication intuitive

O(f(n)) signifie que l'algorithme ne fait pas pire que f(n) a une constante pres, pour n assez grand. On ignore les constantes multiplicatives et les termes de plus petit ordre car ils deviennent negligeables quand n grandit.

Exemple : si un algorithme fait exactement 3n^2 + 10n + 500 operations, on dit qu'il est O(n^2) car pour n grand, le terme 3n^2 domine largement les autres. Les constantes 3, 10, et 500 ne changent pas le "comportement global" quand n tend vers l'infini.

On garde le terme dominant, on ignore les constantes : 3n² + 10n + 500 → O(n²)

Pire cas, meilleur cas, cas moyen

Pire cas (worst case)

Le nombre maximum d'operations pour TOUTE entree de taille n. C'est ce qu'on utilise par defaut avec O() car il donne une garantie : l'algorithme ne fera JAMAIS pire que ca.

Exemple : recherche lineaire dans un tableau de n elements → pire cas = O(n) (element absent ou en derniere position).

Meilleur cas / cas moyen

Meilleur cas : peu interessant en general. Pour la recherche lineaire, c'est O(1) (trouve en premiere position).

Cas moyen : necessite des hypotheses sur la distribution des entrees. Pour la recherche lineaire avec element present et equiprobable → O(n/2) = O(n).

On utilise presque toujours le pire cas pour donner des garanties solides.

Exemples de simplification avec preuve

Exemple 1 : 5n^3 + 2n = O(n^3)

Preuve : pour n ≥ 1, on a 2n ≤ 2n^3. Donc 5n^3 + 2n ≤ 5n^3 + 2n^3 = 7n^3. On prend c = 7 et n0 = 1. Donc 5n^3 + 2n = O(n^3).

Exemple 2 : n^2 + 100n + 10000 = O(n^2)

Preuve : pour n ≥ 100, on a 100n ≤ n^2 et 10000 ≤ n^2. Donc n^2 + 100n + 10000 ≤ n^2 + n^2 + n^2 = 3n^2. On prend c = 3 et n0 = 100.

Exemple 3 : log(n^2) = O(log n)

Preuve : log(n^2) = 2 * log(n). On prend c = 2 et n0 = 1. C'est bien O(log n).

Tri pancake -- analyse de complexite

Principe du tri pancake

On ne peut faire qu'une seule operation : retourner les k premiers elements du tableau (comme retourner une pile de crepes). Pour trier :

  1. Trouver le maximum du tableau (O(n))
  2. Le retourner en premiere position (1 retournement)
  3. Le retourner en derniere position (1 retournement)
  4. Repeter sur le tableau reduit (sans le dernier element)

A chaque iteration, on place un element a sa position finale. On fait n-1 iterations, chacune avec O(n) pour trouver le max + 2 retournements (O(n) chacun). Complexite : O(n^2).

3.3 Theoreme Maitre

Deux formes de recurrence

Forme 1 : T(n) = c * T(n - 1) + b * n^k  -- on reduit la taille de 1 a chaque appel (exemples : fibonacci, factorielle)

Forme 2 : T(n) = c * T(n / d) + b * n^k  -- on divise la taille par d a chaque appel (exemples : recherche dichotomique, tri fusion)

La forme determine quels cas appliquer. Forme 1 → cas 1.1 ou 1.2. Forme 2 → cas 2.1, 2.2 ou 2.3.

Explication intuitive des cas (Forme 2)

On compare le cout de la recursion (c appels) au cout du travail a chaque niveau (n^k, divise par d^k a chaque niveau) :

  • Cas 2.1 (c > d^k) : la recursion domine. Le nombre d'appels croit plus vite que le travail ne diminue. Le cout total est domine par le nombre de feuilles de l'arbre de recursion : O(n^(log_d c)).
  • Cas 2.2 (c = d^k) : equilibre parfait. Le travail a chaque niveau est le meme. Il y a log_d(n) niveaux, chacun coutant O(n^k). Total : O(n^k log n).
  • Cas 2.3 (c < d^k) : le travail domine. Le travail diminue plus lentement que la recursion. Le cout total est domine par le premier niveau : O(n^k).
Tri fusion detaille

T(n) = 2*T(n/2) + O(n) → c=2, d=2, k=1 → dᵈ=2=c → Cas 2.2 → O(n log n)

Explication par l'arbre de recursion :

  • Niveau 0 : 1 probleme de taille n → cout de fusion = n
  • Niveau 1 : 2 problemes de taille n/2 → cout de fusion = 2 x n/2 = n
  • Niveau 2 : 4 problemes de taille n/4 → cout de fusion = 4 x n/4 = n
  • ... chaque niveau coute exactement n operations de fusion
  • Nombre de niveaux : log2(n) (on divise par 2 a chaque fois)
  • Total = n x log2(n) = O(n log n)

Tri a bulles -- O(n^2)

On parcourt le tableau plusieurs fois. A chaque passage, on compare les elements adjacents et on les echange si ils sont dans le mauvais ordre. Les plus grands elements "remontent" vers la fin comme des bulles.

def tri_bulles(T):
    for i in range(len(T)):
        for j in range(0, len(T)-i-1):
            if T[j] > T[j+1]:
                T[j], T[j+1] = T[j+1], T[j]

Tri par insertion -- O(n^2)

On prend chaque element (a partir du 2e) et on l'insere a sa bonne position dans la partie deja triee a gauche, en decalant les elements plus grands vers la droite.

def tri_insertion(T):
    for i in range(1, len(T)):
        cle = T[i]               # element a inserer
        j = i - 1
        while j >= 0 and T[j] > cle:
            T[j + 1] = T[j]      # decaler vers la droite
            j = j - 1
        T[j + 1] = cle           # inserer a la bonne place

Tri par selection -- O(n^2)

On cherche le minimum du tableau non trie, et on l'echange avec le premier element non trie. On repete en reduisant la partie non triee d'un element a chaque iteration.

def tri_selection(T):
    for i in range(len(T)):
        i_min = i                 # indice du minimum
        for j in range(i + 1, len(T)):
            if T[j] < T[i_min]:
                i_min = j
        T[i], T[i_min] = T[i_min], T[i]  # echange

Tri fusion -- O(n log n)

Diviser pour regner : on divise le tableau en deux moities, on trie chaque moitie recursivement, puis on fusionne les deux moities triees en un seul tableau trie.

def fusion(G, D):
    """Fusionne deux tableaux tries G et D en un seul tableau trie"""
    resultat = []
    i = j = 0
    while i < len(G) and j < len(D):
        if G[i] <= D[j]:
            resultat.append(G[i])
            i += 1
        else:
            resultat.append(D[j])
            j += 1
    resultat += G[i:]             # ajouter le reste
    resultat += D[j:]
    return resultat

def tri_fusion(T):
    if len(T) <= 1:
        return T
    mid = len(T) // 2
    G = tri_fusion(T[:mid])      # trier la moitie gauche
    D = tri_fusion(T[mid:])      # trier la moitie droite
    return fusion(G, D)          # fusionner les deux

3.5 Recherche dichotomique

Pourquoi O(log n) ?

A chaque etape, on divise l'espace de recherche par 2. Si on a n elements, apres 1 comparaison il reste n/2, apres 2 il reste n/4, ..., apres k il reste n/2^k. On s'arrete quand n/2^k = 1, soit k = log2(n). Donc au plus log2(n) comparaisons.

Exemple concret : pour n = 1 000 000 elements, log2(1 000 000) ≈ 20. Seulement 20 comparaisons au lieu de 1 000 000 !

# Recherche dichotomique -- O(log n)
def rech_dic(e, T):
    deb, fin = 0, len(T) - 1
    while deb <= fin:
        mil = (deb + fin) // 2
        if T[mil] < e: deb = mil + 1
        else:           fin = mil - 1
    if deb < len(T) and T[deb] == e: return deb
    return -1

3.6 Algorithme de Kadane (sous-somme maximale)

Probleme

Trouver la plus grande somme d'elements consecutifs dans un tableau (qui peut contenir des nombres negatifs). Exemple : [-2, 1, -3, 4, -1, 2, 1, -5, 4] → sous-tableau [4, -1, 2, 1] de somme 6.

Principe de Kadane -- O(n)

On maintient deux variables : cur = meilleure somme se terminant a la position courante, max_s = meilleure somme vue globalement. A chaque element x :

  • cur = max(x, cur + x) -- soit on prolonge la sous-somme courante, soit on en demarre une nouvelle a partir de x
  • max_s = max(max_s, cur) -- on met a jour le maximum global

L'idee cle : si cur + x < x, alors le sous-tableau precedent nous "tire vers le bas" et il vaut mieux repartir de x seul.

def kadane(T):
    max_s = cur = T[0]
    for x in T[1:]:
        cur = max(x, cur + x)
        max_s = max(max_s, cur)
    return max_s

Conversions pleine - creuse

def pleine_vers_creuse(M):
    return [[j for j in range(len(M)) if M[i][j] != 0] for i in range(len(M))]

def creuse_vers_pleine(L):
    n = len(L)
    M = [[0]*n for _ in range(n)]
    for i in range(n):
        for j in L[i]: M[i][j] = 1
    return M

    max_s = cur = T[0]
    for x in T[1:]:

Aide-memoire

Landau : T(n) = O(f(n)) ⇔ il existe c, n0 tels que T(n) ≤ c*f(n) pour tout n > n0

Hierarchie : O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)

DFS = pile = pop() · BFS = file = pop(0)

ABR infixe = valeurs triees · Tas : fils = 2i+1, 2i+2 ; pere = (i-1)//2

Kruskal : trier aretes → retenir si composantes differentes → unifier

Dijkstra : Dist[source]=0, repeter : traiter le sommet avec le plus petit Dist, mettre a jour voisins

Pieges classiques
  • Prefixe (RGD) != Infixe (GRD) != Postfixe (GDR)
  • Tri fusion pas en place (tableau auxiliaire)
  • 2n+1 = theta(2ⁿ) -- le x2 est une constante
  • 3ⁿ - 2ⁿ = theta(3ⁿ) -- pas theta(3ⁿ-2ⁿ)
  • Kruskal : trier d'abord, puis verifier les composantes
  • Bien identifier c, d, k avant d'appliquer le theoreme maitre
  • ABR : l'ordre d'insertion change la forme (equilibre vs degenere)
  • Tas != ABR : un tas n'a pas la propriete gauche < racine < droite
  • Dijkstra ne marche pas avec des poids negatifs
